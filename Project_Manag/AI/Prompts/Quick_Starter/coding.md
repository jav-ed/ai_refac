1. Obvious Code: Make inputs and outputs crystal clear. Avoid unnecessary abstractions - keep methods, classes, and functions straightforward

2. Modular Organization: For each feature, create a main file (descriptive filename with suffix main) alongside a supporting folder. Inside this folder, put all the helper files that the main file uses. For complex tasks that can be seperated in clear subfeatures, this pattern can be applied for deeper lvl organization. This makes the entry point and dependencies immediately clear. It makes it easy to read, understand and expand the code.

3. It can't be stressed enough that folder/file structure should be clear, intuitive. This reinforces point 1 and 2. It says that folder and file naming, their clear relationships, seperated and combined by prupose, having one clear entry point and so on - they all shall enforce the importance of firstly having a clear high level file/folder structure that makes sense, as with this coding afterwards becomes easy.

4. Essential Commenting: Explain the "why" behind code decisions
